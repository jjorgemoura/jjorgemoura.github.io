Functional

At first, functional programming feels like a lot of restrictions:

  • No loops
  • No mutating data
  • No impure functions

One good explanation for my doubts about functinoal programming is:

They boil down to the idea that functional programming is programming with pure functions. Your app is one big pure function. That definition may work well for academics, because functional programs are a means of building formal mathematical proofs. But what about industrial programmers like us? In industrial projects, we do have side effects. We render the user interface, make payments, write to files, use the network, and send emails. These are all side effects. We want our users to be able to do all this stuff. Side effects are the reason why our apps are useful.

Having side effects is necessary.

Thus, what we want (or could achive) is to have as many lines of code as possible in pure functions and as few as possible in side effects.

The essence of functional programming is that it makes the distinction between pure code and impure code and minimizes the latter.

The core of any functional programming relies on these 3 concepts:

In a nutshell, a functional program can be broken into three parts: functions, effects and data.

We can compare these concepts with Object-oriented programming languagues' own concepts, encapsulation, inheritance and polymorphism.

You don’t want to use functional programming for websocket communication layer. Conversely, Fourier transform is naturally described with functions instead of objects and classes.

Functions

In FP, by Function we mean, pure function.

A pure function can be defined as:

  • cannot depend on anything that changes.
  • doesn’t change anything by itself.
  • can be called many times and it will return the same result every time.

Effects

Unlike functions, calling an effect many times does not have the same result.

Effects depend on how many times they are called and when they are called.

For example, reading from a global or a mutable variable is another example of an effect, because it depends on when it happens. It is possible to have different values at different moments in time. Effects are hugely important because is where our code meets the real world.

However, effects are the most troublesome part of a program, because it makes a difference when, in what order, and how many times you run them.

Data

Data is information about an entity, represented in the memory. Data requires interpretation.

Data is represented as a type. Types are important because they define the kinds of computation allowed on certain data so that data knows how to interpret itself.

Reading from a mutable data structure is an effect. Therefore, in functional programming we strive to make use of immutable data structures.

Is Swift Functional?

No.

Swift has some features (map, reduce, first-class functions, pattern matching, etc) from functional languages. However, having those features doesn't mean we call it a full Functinal Language. Swift lacks, by default, much more functional features.

Some languages make it difficult to run an effect. For instance, Haskell has a special parameterized type called IO. Any value in an IO context must stay in this context. This doesn’t let you mix pure code with impure.

There is no rule that says we can do hybrid development, using OOP where best fits and Functional programming where makes sense.
