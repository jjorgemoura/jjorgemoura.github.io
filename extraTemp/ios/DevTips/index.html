#1 No Raw Loops

Sean Parent (a.k.a. "Crusty's cousing"), C++ Seasoning / WWD 2018

>If you want to improve the code quality in your organization, replace all of your coding standards with one goal, No Raw Loops.

We should not write any loop code in our codebase. We should use a known function. If such function doesn't exists yet, let's create it.

Bad

// Remove each selected item.
// ....
for i in (0..<shapes.count).reversed() {
    if shapes[i].isSelected {
        shapes.remove(at: i)
    }
}

Good

shapes.removeAll(where: { $0.isSelectedb })

This can be even smaller shapes.removeAll { $0.isSelectedb }.

#3 LLDB: setup on Xcode

This is a really important degug tip. Xcode debugger (lldb) doesn't work 100% out of the box. It requires some setup. I will not enter in the details of why this happens. There is plenty of info about this out there. Let just focus on the fix/setup.

When we try to inspect a variable in the Xcode's console and get the famous error: property "xpto" not found on object of type "XPTO".

The default context is still Objective-C. We need to set Swift. We need to setup the lldb properly. So, we need to set:

expr -l swift -O -- UIApplication.shared

An alternative way to overcome this issue is:

expr @import UIKit
expr @import Foundation

In order to automate this setup is through the use of a breakpoint that execute always tbis command and that continue execution.

#4 LLDB: General Purpose Registers

Carola Nitz, App Builders 2018, 2018 (youtube)

Run the next command in Xcode's console:

register read

An equivalent way to achive the same information is through a launch argument.

-UIViewLayoutFeedbackLoopDebuggingThreshold 100 // 50 to 5000
