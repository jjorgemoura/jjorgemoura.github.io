iOS Architectural Questions

Main reference:

https://www.youtube.com/watch?v=ZgdS0EUmn70&t=1091s

Architectural interviews are much more wide in terms of spectrum, of competence, of speciality, of breadth, of depth. There is no right or wrong answer. There is always some tradeoffs. We need to understand a complex bunch of different things. There is, though, always a set of horrible answers.

Tip:

Interview structure: Fuzzy problem with a few (super important) details. Also, there will be a very specific constraints to satisfy.

Tip:

These kind of interviews are useful to look at Technical Leadership skills.

Notes:

I can explore my process in the past, when all the system architecture and design started with a ER model in the database.

Specific Example:

Build, for a very successful multi-platform app, with 500M users per month, a logging system in the app to send debug information to our backend.

Initially we should explore all the information that is given to us. Question that information.

For example, what mean 500M users a month? for example, this number doesn't tell us anything. At least, something important or relevant for our problem. I would say the important aspect for our system is average users at same time (peak traffic). This will tell us the load that our backend should handle. Probably will be around 68k concurrently at peak time. Can the server handle this load of users?!

Some topics expected to be covered:

  • battery consumption
  • cellular data consumption
  • on device implications
    • how to storage while offline
    • will interrupt UI Thread
    • how to send all the information (API strategy)
    • time frequency or by event
    • server implications in terms of resources and workload.
    • how much more work load is this compared with the present situation. Is this just a new endpoint among 40 others (so a minor increase) or is this like an increase of 25% of endpoints?
    • how much data (size) can we transmit to the server with:
    • the user notice and realise it is to much,
    • the server provider (network provider) complaint regarding the excessive traffic
    • the server provider realise how much storage it will be needed
    • how data is transmitted:
    • encrypted?
    • compressed?
    • protocol?
    • stored locally (and for how long)
    • if offline, how to synchronise data (and be sure nothing is lost)
    • how to support different platforms
    • ios
    • android
    • windows phone
    • apple TV
    • chromecast
    • smart TVs
    • etc
    • Technology stack
    • on client
    • on server
    • can we send, or should, info via mobile data or WiFi only
    • we can also explore how long (resources) will take this to be built
    • all team
    • resources
    • maintenance cost
    • how to access this logs (by whom, from where)
    • data privacy (in different countries)

Basically, the idea behind is talk about high scale systems.

Tip:

Break down the problem. Focus initially on the things or areas we are more comfortable with. After, address the other areas. Is the way we face the other areas that show our capabilities when face something new and challenging.

Make steady progress through the fuzzy problem, with approximations that are like reasonable.

Keep going explore all the details, don't give up. (Winston Churchill once say, "If you are going through hell, keep going!")

Ted Talk face this as a talk where you present to the other person what vision about this problem. Could be a kind of elevator pitch.

Probably a good training tip, or demonstration in a real interview, is using mental maps to write down all the bits.

Tip:

Try to engage with the interviewer. The idea, as I already know, is to see if the person is compatible with the team, if the team would like to work every day with this new potential member. A bad attitude is a very bad signal.

Final Points

  • The goal of these interviews are, no matter the level we are, or how far we are in the carreer, designed to find the interviewee's edges and boundaries. So a to easy interview is a bad sign. Probably a very hard interview is also a bad sign. It should be like a eye test, you want to end with the correct level.
  • When we achieve a big architectural win, we need to talk about then. We must to talk about them. We should talk about them:
    • internally
    • in blogs
    • in conferences
    • Read and follow every technical information (blogs, presentations, etc) and people from the company you want to join in order to be exposed to their ecosystem, challenges, achievements, limitations, etc.
    • The tech industry will hire any person with passion on that domain to be awesome. So, we need to be passion on something in orders of magnitude that will make us a "reference" in that subject.
