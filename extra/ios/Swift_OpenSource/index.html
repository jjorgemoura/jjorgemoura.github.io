LSP - Language Server Protocol

It's a JSON specification about the communication between client (an independent IDE) and a server (the language provider).

The server is the provider. In reality, could be an external webserver, a local or remote socket or even a same thread call.

Three types of communications:

  • request
  • response
  • notifications

A request always get a response. The notifications no.

When a user edits one or more source code files using a language server protocol-enabled tool, the tool acts as a client that consumes the language services provided by a language server. The tool may be a text editor or IDE and the language services could be e.g. refactoring, code completion, etc.

The client informs the server about what the user is doing, e.g. opening a file, inserting a character at a specific text position, etc. The client can also request the server to perform a language service, e.g. to format a specified range in the text document. The server answers a client's request with an appropriate response. For example, the formatting request is answered either by a response that transfers the formatted text to the client or by an error response containing details about the error.

The use of LSP tools allows to decouple the text-editor (IDE) from the Language Service (language stack, such as text formatter, syntax highlight, compiler, etc)

DSL - Domain Specific Language

DSL is on opposite of GPL - General Purpose Language.

Rather than being complete stand-alone languages - like Swift is - DSLs are often hosted in other languages, and as such, need to use a grammar that's also perfectly valid in their host language.

A very clear example is the syntax of a Cocoa Pods' Pod file, the Fastlane's fastfile, Swiftlint's yaml file. Those syntaxes are basically DSL.

All graphical APIs (SnapKit, Apple's VFL, etc) are considered DSL, a domain specific way of writing some code that later will be "processed" by the main language.
